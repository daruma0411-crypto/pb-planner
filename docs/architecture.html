LLM / Claude Sonnet 4

Large Language Model — 本システムでは Anthropic 社の Claude Sonnet 4 を採用

大量のテキストデータから言語パターンを学習したAI。人間のように文章を理解し、生成できます。モデルにはサイズ違い（Haiku＜Sonnet＜Opus）があり、Sonnet はコストと性能のバランス型です。

たとえ：世界中の本・論文・Webを読んで育った超博識な相談相手。ChatGPTの「GPT-4」に相当するもので、車でいえば車種名（カローラ、プリウス等）のようなもの。

たとえ：新人社員への研修マニュアル。「あなたはPB企画のプロです。お客様にはこう対応してください」という業務指示書。

たとえ：秘書が「調べておきますね」と言って自分で資料室に行って調べてくるイメージ。

Application Programming Interface

ソフトウェア同士が通信するための「窓口」。本システムはClaude APIを通じてAIにテキストを送り、回答を受け取ります。

たとえ：レストランの注文カウンター。メニュー（仕様）に従って注文（リクエスト）すると料理（回答）が出てくる。

JSONL（製品データ形式）

JSON Lines

1行に1製品の情報が入ったテキストファイル。Excelの1行＝1製品と同じ考え方で、プログラムが高速に読み書きできます。

たとえ：1行1枚の製品カード。Excelシートの行がそのままテキストファイルになったもの。

一般的な RAG との違い — 本システムの工夫

  • 検索だけでなく「行動」する — 一般的なRAGは「検索→回答」の1方向。本システムはFunction Callingにより、検索・PBカード更新・Excel生成・分析チャート描画までAIが自律的に実行します
  • ベクトル検索不要の軽量設計 — 一般的なRAGはEmbedding+ベクトルDBが必須。本システムは製品数が数百件規模のため、JONLファイルの全文検索で十分高速に動作し、インフラコストを大幅に削減しています
  • 構造化データをそのまま活用 — RAGは非構造テキストからの断片検索が主流。本システムは製品スペック（型番・価格・寸法等）を構造化JSONで保持し、正確な数値比較・フレームワーク分析を可能にしています
  • マルチステップの業務フロー — RAGは1問1答が基本。本システムは「検索→選定→分析→仕様調整→ドキュメント生成」の5段階ワークフローを1つのチャットで完結させます

JSONL vs CSV — 実際の製品データで比較

同じ製品データを2つの形式で表現した例です。JONLは「入れ子構造」を自然に扱えます。

✅ JSONL（本システムで採用）— 1行 = 1製品
{
  "maker": "トミー精工",
  "model": "FLS-1000",
  "name": "高圧蒸気滅菌器 FLS-1000",
  "price_numeric": 1080000,
  "features": {
    "ローテーブル設計": "テーブルトップ高さ79cm",
    "大口径チャンバー": "φ45cm×100L"
  },
  "specs": {
    "缶体容量": "100リットル",
    "本体質量": "108kg",
    "消費電力": "3.5kW"
  }
}
CSV（一般的な形式）— 同じデータを表現すると…
# 入れ子構造（features, specs）が表現しにくい
# カラム数が爆発し、読みづらくなる

maker,model,name,price_numeric,feature_1_name,feature_1_val,feature_2_name,feature_2_val,spec_缶体容量,spec_本体質量,spec_消費電力
トミー精工,FLS-1000,高圧蒸気滅菌器 FLS-1000,1080000,ローテーブル設計,テーブルトップ高さ79cm,大口径チャンバー,φ45cm×100L,100リットル,108kg,3.5kW

# 問題点:
# ・製品ごとにスペック項目数が異なると列が合わない
# ・features の数が可変だと列数が定まらない
# ・入れ子(nested)データを扱えない

7 セキュリティ・制限事項

安全に運用するための設計上の考慮事項です。

🔒 セキュリティ対策

  • APIキーは環境変数で管理（コードに埋め込まない）
  • セッションは1時間で自動削除
  • 古いセッションファイルは100件超で自動クリーンアップ
  • HTTPS通信（Railway標準）

⚠️ 既知の制限

  • 同時利用 — ワーカー1のため、同時に1ユーザーのみ処理
  • AI応答時間 — 複雑な分析は最大120秒かかる場合あり
  • データ規模 — 大量製品の全文検索には非対応（現状数百件規模）
